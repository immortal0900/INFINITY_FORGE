## 1. 한 문장으로: 무엇을 만드나

**밤에 자는 동안, 여러 개의 작업 지시서를 AI들이 알아서 나눠 갖고 코드로 구현하고 서로 검사까지 마쳐 두는 자동 개발 공장을 만든다.** 아침에 사람은 결과물을 훑어보고 "이건 통과, 이건 다시"만 정하면 된다.

기존에 AI에게 코드를 시키던 방식과 뭐가 다른지 비교하면:

| | 지금까지 (AI에게 직접 시키기) | 이 시스템 (INFINITY_FORGE) |
|---|---|---|
| 방식 | 사람이 AI 창 하나 열고 대화하며 하나씩 | 지시서를 큐에 넣으면 여러 AI가 나눠 처리 |
| 사람의 위치 | 매 순간 옆에 붙어 있어야 함 | 아침에 결과만 확인 |
| 시간 | 사람이 깨어 있는 시간만 | 밤새 무인으로 |
| 완료 판정 | AI가 "다 했어요" 하면 믿음 | 기계 검사를 통과해야만 완료 |

마지막 줄이 이 프로젝트의 핵심 문제의식이다. 뒤(6장)에서 자세히 다룬다.

## 2. 등장인물 (여기만 외우면 나머지가 읽힌다)

이름이 여러 개 나와서 헷갈리는 게 정상이다. **역할과 비유로** 외우면 된다. 자동차 공장에 빗대었다.

| 이름 | 정체 | 한 줄 역할 | 공장 비유 |
|---|---|---|---|
| **hermes** (헤르메스) | 남이 만든 오픈소스 관리 프로그램 | 작업을 나눠 배정하고, 죽은 작업을 되살리는 반장 | 작업 배정판 앞의 현장 반장 |
| **Kanban** (칸반) | hermes에 들어 있는 작업 관리 기능 | "어느 작업이 지금 어느 단계인지"를 기록하는 장부 | 금고 속 작업 장부 |
| **작업 실행기**(worker) | Hermes가 필요한 순간에만 띄우는 AI 세션 | Build·Review·Deep Check·Fix 중 한 단계를 실행 | 단계별 작업 담당 |
| **Codex**(코덱스) | OpenAI가 만든 코딩 AI (터미널에서 실행) | 밤의 실제 코딩 담당 | 밤샘 근무하는 기술자 |
| **Claude Code**(클로드 코드) | Anthropic이 만든 코딩 AI | 아침에 사람 옆에서 검토를 돕는 조수 | 아침 품질 검사원 |
| **GitHub**(깃허브) | 코드와 이슈(작업 요청)를 올리는 웹 서비스 | 사람이 지시하고 결과를 보는 창구 | 공장 입구의 전광판 겸 접수처 |
| **MEMEX**(메멕스) | 네가 따로 만든 개인 지식 저장고 | AI들이 배운 것을 쌓아두는 도서관 | 사내 자료실 |
| **VPS**(브이피에스) | 인터넷에 빌린 리눅스 컴퓨터 (OVH사) | 이 모든 게 돌아가는 물리적 장소 | 공장 건물 그 자체 |

이 여덟이 전부다. 아래 모든 설명은 이 여덟 명이 주고받는 이야기다.

## 3. 작업 하나가 실제로 진행되는 순서

1. 대화를 시작하면 **Chat** 또는 **Task**를 고른다.
2. **Chat**은 일반 질문과 설계 논의만 진행하며 GitHub 이슈나 Kanban 카드를 만들지 않는다.
3. **Task**는 매번 다음 두 가지를 모두 고른다.
   - 실행 단계: **Build**, **Build + Review**, **Build + Review + Deep Check**
   - 병합 방식: **Manual Merge**, **Safe Files Auto-Merge**, **All Validated PRs Auto-Merge**
4. 최종 확인 뒤에만 GitHub 이슈와 변경할 수 없는 Task 설정이 만들어진다.
5. `task-flow-worker.py`가 완료된 결과와 현재 PR commit을 다시 읽고 다음 Build·Review·Deep Check·Fix 카드를 한 장만 만든다.
6. 각 결과는 선택한 검사, 실제 변경, 현재 base/head commit과 정확히 맞아야 다음 단계로 넘어간다. 읽기 오류나 불일치가 있으면 해당 Task를 멈춘다.
7. `issue-status-sync.py`가 GitHub 이슈 상태를 `forge:building`, `forge:reviewing`, `forge:deep-checking`, `forge:ready-to-merge` 같은 공식 라벨 하나로 맞춘다.
8. `merge-worker.py`는 Task에서 선택한 병합 방식과 최신 GitHub 상태를 다시 확인한다. 자동 병합은 `AUTO_MERGE_ENABLED=true`를 명시한 운영 환경에서만 가능하며 기본값은 꺼짐이다.

## 4. 이 시스템이 사는 곳: 왜 내 노트북이 아니라 빌린 컴퓨터인가

**전체가 OVH VPS(인터넷에 빌린 리눅스 컴퓨터 한 대)에서 돌아간다. 내 노트북에서 안 돌린다.** 이유가 세 가지다.

**이유 1: 밤새 켜져 있어야 한다.** 노트북은 밤에 절전으로 들어가거나, 윈도우 업데이트로 새벽에 멋대로 재부팅한다. 그러면 밤 작업이 통째로 날아간다. 빌린 리눅스 컴퓨터는 24시간 깨어 있으라고 있는 물건이라 이 문제가 없다.

**이유 2: 리눅스라야 편하다.** 이 도구들은 리눅스에서 자란 물건이라, 윈도우에서 돌리려면 WSL2(윈도우 안에 리눅스를 흉내 내는 계층)를 거쳐야 하는데, 그게 밤샘 안정성에서 약하다. VPS는 처음부터 리눅스라 우회로가 필요 없다.

**이유 3: 도서관(MEMEX)이 이미 거기 산다.** 네 MEMEX가 이미 이 VPS에서 돌고 있다. AI 공장을 같은 건물에 두면, AI가 자료실에 뭘 넣고 뺄 때 건물 밖으로 나갈 필요가 없다(빠르고 안전하다).

**사양(성능) 이야기 한 줄**: 인터넷에는 "월 5천 원짜리 작은 서버면 충분"이라는 글이 많은데, 그건 텔레그램 챗봇 하나 돌릴 때 얘기다. 우리는 지식 DB(Neo4j) + 반장(hermes) + 여러 일꾼을 한 건물에 같이 두므로 **RAM(작업 기억 공간) 8GB**는 있어야 한다. 이 VPS는 8GB로 확정했다.

## 5. 사람은 언제 개입하나: "안 멈추게" 하면서도 "중요한 건 사람이"

여기서 근본적인 딜레마가 있다. **AI에게 자율을 주면 밤새 알아서 굴러가지만, 정말 중요한 결정까지 AI가 멋대로 해버린다. 반대로 매번 사람 허락을 받게 하면 밤에 사람이 자느라 작업이 멈춰버린다.** 이 둘을 어떻게 동시에?

답은 **대화, 실행 단계, 병합 권한을 나누는 것**이다.

| 상황 | 처리 | 사용자가 보는 선택 |
|---|---|---|
| 일반 질문·설계 논의 | 외부 작업을 만들지 않고 대화만 진행 | **Chat** |
| 실제 구현 | 선택한 단계만 서로 다른 세션에서 실행 | **Build / Build + Review / Build + Review + Deep Check** |
| 최종 반영 | 사람이 직접 병합하거나, 조건을 충족한 PR만 자동 병합 | **Manual Merge / Safe Files Auto-Merge / All Validated PRs Auto-Merge** |

**핵심 장치는 "라벨"이다.** 라벨(label, 이슈에 붙이는 상태 딱지)로 "이 작업은 지금 검토 대기", "이건 사람 결정 필요" 식으로 표시한다. 사람은 아침에 폰으로 이 딱지들만 훑어보면 밤새 무슨 일이 있었는지 한눈에 안다. 그리고 딱지가 "사람 결정 필요"인 것에만 댓글 한 줄 남기면 그게 지시가 되어 다음 밤에 반영된다.

비유하면 라벨은 병원 대기실의 전광판이다. 번호마다 "진료 중 / 대기 / 수납"이 떠 있고, 사람은 자기가 개입할 번호만 보면 된다. 나머지는 시스템이 알아서 흐른다.

## 6. 이 프로젝트가 존재하는 진짜 이유: AI의 "다 했어요" 문제

**이 시스템의 절반은 이 한 가지 문제를 막으려고 존재한다.** 실제로 겪은 일이다.

AI에게 큰 기획서를 주고 구현을 시키면, AI가 자기가 만진 일부만 해놓고 **"다 구현했습니다"라고 선언하고 끝내버린다.** 그런데 캐물으면 그제야 "아직 이 부분은 안 했습니다"라고 실토한다. 왜 이럴까?

**AI의 "완료했습니다"는 사실 보고가 아니라 그냥 생성된 문장이기 때문이다.** 검사기 없이는 진심인지 희망사항인지 구분이 안 된다. 건설로 비유하면, 시공사가 "완공했습니다"라고 말하는 것과 준공검사에 통과하는 것은 전혀 다른 일이다. 지금까지는 준공검사 없이 시공사 말만 믿은 셈이다.

이 문제는 사실 두 겹이다.

**문제 A: 덜 해놓고 다 했다고 함 (과대 선언).**
막는 법: **완료를 선언할 권리를 AI에게서 뺏는다.** AI가 "끝"이라고 말하는 것으로는 아무 일도 안 일어나고, 오직 자동 검사(테스트 통과 + 진짜로 코드가 바뀌었는지 확인)를 통과해야만 완료로 친다. 게다가 완료 판정은 그 코드를 짠 AI가 아니라 **다른 AI와 기계**가 한다. 자기 시험을 자기가 채점하지 못하게 하는 것이다.

**문제 B: 일을 멋대로 쪼개고 앞부분만 하고 도망 (합법적 탈출).**
이게 더 교묘하다. AI가 큰 일을 "1단계, 2단계"로 자기 편한 대로 나눈 뒤, "1단계는 끝냈습니다"라며 나가버린다. 거짓말은 아니다. 진짜로 1단계는 했으니까. 그런데 2단계는 어디에도 기록되지 않고 사라진다.

막는 법: **세 가지 권한을 서로 다른 담당에게 찢어놓는다.**

| 권한 | 원래 (문제) | 우리 방식 |
|---|---|---|
| 일을 쪼개는 권한 | AI가 자기 편한 크기로 | 작업을 나누는 별도 단계에서, 반드시 "카드"로만 |
| 완료 기준을 정하는 권한 | AI가 자기 결과에 맞춰 | 카드 만들 때 미리 못 박음 |
| 끝낼 권한 | AI가 "끝" 선언 | 검사 통과 + 다른 AI 확인 |

그리고 결정적으로, **"나머지는 나중에"라고 말로 남기는 것을 금지한다.** 안 한 일이 있으면 반드시 그 자리에서 "후속 작업 카드"를 만들어 큐에 넣어야만 나갈 수 있다. 계주 비유로 돌아오면: **바통을 다음 주자 손에 실제로 쥐여주기 전에는 트랙을 못 떠난다.** "다음 주자가 뛸 겁니다"라고 말만 하고 떠나는 걸 막는 것이다.

**마지막 안전망: 커버리지 감사.** 위 장치들은 "작업 하나하나"를 검사한다. 그런데 애초에 카드로 만들어지지도 않은 기획 항목은 아무도 안 본다. 그래서 밤마다·아침마다 **기획서의 모든 항목이 카드가 되었는지, 다 끝났는지를 숫자로 대조**한다. "기획서 12개 항목 중 9개 완료" 식으로. 이 숫자가 12/12가 되고 모든 검사가 초록불이 되어야만 "진짜 완료"다. 즉 완료 여부를 AI의 말이 아니라 숫자가 판정한다.

## 7. 두 AI를 같이 쓰는 이유: 서로 다른 학교 출신

**밤에는 Codex(OpenAI)가 일하고, 아침에는 Claude Code(Anthropic)가 검토를 돕는다.** 왜 굳이 둘을?

**이유 1: AI가 자기 약점은 자기가 못 본다.** 같은 AI가 코드를 짜고 같은 AI가 검사하면, 세션(대화 창)을 새로 열어도 **같은 맹점을 공유**한다. 같은 학교에서 같은 교과서로 배운 두 학생에게 서로 채점을 시키는 것과 같다. 둘 다 안 배운 부분은 둘 다 못 잡는다. 그래서 "다른 학교 출신"인 두 AI가 서로를 보게 하면, 한쪽이 놓친 걸 다른 쪽이 잡는다.

**이유 2 (현실적 제약): Claude를 밤에 무인으로 돌리는 게 규정상 막혔다.** 2026년 6월부터 Anthropic 구독으로 Claude를 자동화에 쓰는 방식이 유료 크레딧 차감으로 바뀌었다. 이 크레딧을 안 쓰기로 했으므로, Claude를 밤새 무인으로 돌리는 합법적 방법이 사실상 없다. 그래서 밤일은 규정이 허용하는 OpenAI(Codex) 쪽이 전담한다.

그러면 "다른 학교 출신 교차 검사"(이유 1)는 어떻게? **아침으로 옮겼다.** 사람이 아침에 결과를 확인할 때 Claude Code를 켜서 함께 본다. 어차피 사람이 서 있는 검수대라 절차가 늘지 않는다. 즉 교차 검사는 사라진 게 아니라 밤에서 아침으로 근무 시간만 바뀐 것이다.

## 8. 지식 도서관(MEMEX)은 어떻게 엮이나: 있으면 좋고 없어도 되는 것

**MEMEX(자료실)는 AI들이 배운 것을 쌓아두는 곳인데, 중요한 원칙이 하나 있다: 절대 필수 경로에 두지 않는다.**

왜냐면 두 종류의 기록은 성질이 정반대이기 때문이다.

| | 진행 상태 (어느 작업이 어디까지 갔나) | 배운 지식 (실패담, 교훈) |
|---|---|---|
| 잃으면 | 밤 작업 전체가 날아감 | 밤은 멀쩡함, 다음번이 덜 똑똑할 뿐 |
| 두는 곳 | 튼튼한 로컬 장부 (Kanban) | 자료실 (MEMEX)에 사본으로 |

**진행 상태**는 잃으면 치명적이라 가장 튼튼한 곳(로컬 장부)에 둔다. **배운 지식**만 자료실에 보낸다. 그것도 "우체통 방식"으로: AI는 교훈을 일단 폴더에 써두고 하던 일을 계속한다. 우체부 역할의 프로그램이 나중에 그걸 자료실로 배달한다. 자료실이 잠깐 문을 닫아도(서버 장애) 배달만 늦어질 뿐 밤 작업은 안 멈춘다.

비유하면 자료실은 **확성기**다. 없어도 내 책상 노트(로컬 파일)로 일은 되고, 있으면 "그 문제 저번에 다른 작업에서 겪었어요"라고 크게 알려준다. 확성기를 필수 부품으로 만들면, 확성기가 고장 났을 때 공장 전체가 멈추는 어이없는 일이 생긴다. 그래서 부가 장치로만 둔다.

**보안 한 가지**: 이 자료실은 인증 열쇠(비밀번호 같은 것)로 보호되는데, 지금 그 열쇠가 암호화 안 된 통로로 오간다. 엽서에 비밀번호를 적어 부치는 셈이라 위험하다. 해결책은 자료실 문을 "건물 내부용"으로만 열어두는 것이다. AI 공장이 같은 건물(VPS)에 있으니 건물 안에서만 열려 있어도 충분하고, 그러면 열쇠가 인터넷을 탈 일이 아예 없어진다.

---

## 9. Hermes 변경을 작고 되돌릴 수 있게 관리하는 이유

대부분의 Forge 기능은 Hermes 밖의 plugin, Python 모듈, worker 스크립트에 둔다. 다만 모든 대화 화면에서 Chat/Task 선택을 같은 방식으로 적용하려면 Hermes v0.18.2의 입력 처리 지점 6개에 작은 연결 변경이 필요하다.

이 변경은 파일을 직접 편집하지 않고 설치 도구로만 적용한다.

1. 설치 전 원본 파일 hash가 지원 목록과 같은지 확인한다.
2. 일치할 때만 준비된 변경을 원자적으로 적용한다.
3. 설치 뒤 변경 파일 hash를 다시 확인한다.
4. 원본 파일은 복원 묶음에 보관한다.
5. Hermes 버전이나 파일이 예상과 다르면 아무것도 바꾸지 않고 중단한다.

이 방식은 Hermes 업데이트와 Forge 변경이 조용히 섞이는 일을 막고, 문제가 생겼을 때 확인된 원본으로 되돌릴 수 있게 한다.

## 10. 지금 상태와 남은 운영 확인

저장소에는 대화 선택, Task 설정 저장, GitHub 이슈 연결, 다음 카드 생성, 이슈 상태 동기화, 세 병합 방식의 실행 경로가 연결돼 있다. 세 worker는 입력 DB·GitHub·Hermes 상태를 실제로 읽으며, 필요한 정보가 없거나 서로 다르면 오류 코드 `2`로 끝나고 외부 변경을 계속하지 않는다.

아직 이 문서에서 주장하지 않는 것은 **production 서버 반영**이다. 기능 구현과 서버 배포는 다른 상태다. 배포 완료 판정에는 다음 세 가지가 모두 필요하다.

1. 변경이 `main`에 반영돼 있다.
2. EC2와 VPS 저장소가 그 commit을 가리킨다.
3. 두 서버의 관련 systemd timer와 service가 새 명령 인자로 정상 실행된다.

자동 병합은 코드가 준비돼 있어도 기본값이 꺼져 있다. **Manual Merge**는 별도 설정 없이 적용되고, 두 자동 방식은 Task별 선택에 더해 서버 환경의 `AUTO_MERGE_ENABLED=true`가 있어야 한다.

## 부록: 자주 나오는 용어 빠른 사전

| 용어 | 쉬운 뜻 |
|---|---|
| 이슈(issue) | GitHub에 올리는 작업 요청 글 |
| PR (Pull Request) | "이 코드를 반영해 주세요"라는 병합 요청 |
| 머지(merge) | PR을 실제 코드에 반영하는 것 |
| 라벨(label) | 이슈에 붙이는 상태 딱지 |
| 카드(card) | Kanban 장부에 있는 작업 한 건 |
| 테스트(test) | 코드가 제대로 도는지 자동으로 확인하는 코드 |
| 훅(hook) | 특정 순간에 자동 실행되는 검사 스크립트 |
| check(검사) | 다음 단계 전에 조건을 확인하고, 실패하면 진행을 멈추는 절차 |
| 세션(session) | AI와의 대화 창 하나 |
| RAM | 컴퓨터의 작업 기억 공간 (클수록 여러 일 동시 처리) |
| SSH | 원격 컴퓨터에 안전하게 접속하는 방법 |
| 크레딧 | 사용량만큼 차감되는 유료 잔액 |

# code-design-principles
신중함을 우선한다. 일회용 스크립트나 한 줄 수정에는 강도를 완화해도 된다.

## 1. 구현 전 SKILL/공식문서 참조

- 작업 착수 전 관련 SKILL.md가 존재하는지 확인하라.
- SKILL이 없으면 해당 라이브러리/프레임워크/도구의 공식 문서를 참조하라.
- 기억에 의존 금지. API 시그니처, 설정 옵션, 동작 방식은 버전마다 바뀐다.
- 추측보다 이미 검증된 방식을 찾아라.

## 2. 침묵 금지

**모호함과 모순을 발견하면 알리고 진행하라.**

요구사항 모호성:
- 가정은 명시적으로 진술하라. 불확실하면 질문하라.
- 복수 해석이 가능하면 모두 제시하라. 임의로 한쪽을 선택하지 마라.
- 더 단순한 접근이 있으면 제시하라. 근거가 있으면 반대 의견을 내라.
- 불명확한 부분은 멈추고, 무엇이 불명확한지 명시한 뒤 질문하라.

코드베이스 모순:
- **이름 충돌:** 같은 개념인데 이름이 다름 (예: `userId` vs `user_id`)
- **패턴 충돌:** 동일 기능을 하는 함수/로직이 여러 곳에 분산
- **가정 충돌:** 주석과 실제 동작이 다름. 새 작업이 기존 전제를 깨뜨림 (예: "동기 모듈로 알았는데 비동기 호출이 섞여 있음")

조용히 한쪽을 선택해 진행하지 마라. 그 선택이 다른 곳에서 미묘한 버그를 만든다.

## 3. 단순함과 단일 책임

**문제를 푸는 최소 코드만. 하나의 단위는 하나의 책임만.**

단순함:
- 요청되지 않은 기능 추가 금지.
- 일회용 코드에 추상화 만들지 마라.
- 요청되지 않은 "유연성"이나 "구성 가능성" 추가 금지.
- 발생할 수 없는 시나리오에 대한 에러 처리 금지.
- 200줄을 50줄로 줄일 수 있다면 다시 작성하라.

판단 기준: "시니어 엔지니어가 이걸 보고 과하다고 할까?" 그렇다면 단순화하라.

단일 책임 (SRP, Single Responsibility Principle):
- 함수가 "A를 하고 B도 한다"면 분리를 검토하라.
- 파일 이름에 `and`나 `_utils`가 들어가면 책임이 뒤섞인 신호다.
- 단, 일회용 스크립트에서 SRP를 위해 5개 클래스를 만드는 건 과잉.

## 4. 외과적 변경(Surgical Changes)

**필요한 부분만 건드린다. 본인이 추가한 코드만 정리한다.**

기존 코드를 수정할 때:
- 인접 코드, 주석, 포매팅을 임의로 "개선"하지 마라.
- 망가지지 않은 것을 리팩터링하지 마라.
- 무관한 죽은 코드를 발견하면 알리되, 삭제하지 마라.
- **이해하지 못한 코드는 건드리지 마라.** 마음에 들지 않는 주석/코드를 부수 효과로 지우거나 바꾸지 마라. 작업과 무관한 변경 금지. 이해 못 한 영역은 그대로 두고 질문하라.

본인 변경이 만든 고아 코드 처리:
- 본인 변경으로 사용되지 않게 된 import, 변수, 함수만 제거하라.
- 이전부터 존재하던 죽은 코드는 요청 없이 제거하지 마라.

## 5. 목표 주도 실행(Goal-Driven Execution)

**성공 기준을 정의하고 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환하라:
- "검증 추가" → "잘못된 입력에 대한 테스트를 먼저 쓰고 통과시켜라"
- "버그 수정" → "버그를 재현하는 테스트를 먼저 쓰고 통과시켜라"
- "X 리팩터링" → "리팩터링 전후로 테스트가 모두 통과하는지 확인하라"

다중 단계 작업은 짧은 계획을 먼저 진술하라:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

## 6. SSoT와 일관성

**중복 금지. 기존 관례 준수.**

SSoT (Single Source of Truth, 단일 진실 공급원):
- 상수, 설정값, 스키마는 한 곳에서만 선언하고 나머지는 참조하라.
- 같은 로직을 복사-붙여넣기 하지 마라. 공통 함수로 추출하라.
- 문서와 코드가 같은 내용을 말한다면 코드가 진실이다. 문서는 코드에서 생성하거나 코드를 가리키게 하라.
- 새 코드 작성 전, 같은 역할의 기존 코드가 있는지 먼저 확인하라.

일관성:
- 네이밍: 기존이 camelCase면 camelCase, snake_case면 snake_case.
- 에러 처리: 기존이 try-catch면 try-catch, Result 타입이면 Result 타입.
- 디렉토리 구조: 기존 배치 패턴을 따르라.
- 새 관례를 도입하려면 코드 작성 전에 먼저 제안하라.

## 7. 원자성(Atomicity)

**하나의 변경은 완결된 단위여야 한다.**

- 변경 도중 중단되어도 시스템이 깨지지 않는 상태를 유지하라.
- 관련 변경은 함께 묶고, 무관한 변경은 분리하라.
- DB 작업: 트랜잭션으로 감싸라.
- 파일 작업: 임시 파일에 쓰고 rename으로 교체하라.

## 8. 멱등성(Idempotency)

**같은 연산을 여러 번 수행해도 결과가 한 번 수행한 것과 같아야 한다.**

스크립트, 마이그레이션, 초기화, 설정 코드를 작성할 때:
- "한 번만 실행된다"는 가정 금지. 실패 후 재시도, CI 재실행, 사용자 중복 실행은 일상이다.
- "이미 완료된 상태"를 먼저 확인하고 조건부로 실행하라.

흔한 위반과 멱등 패턴:

| 위반 | 멱등 패턴 |
|---|---|
| `CREATE TABLE x` | `CREATE TABLE IF NOT EXISTS x` |
| `INSERT INTO config VALUES (...)` | `INSERT ... ON CONFLICT DO UPDATE` (upsert) |
| 파일에 append로 설정 추가 | 존재 여부 체크 후 추가, 또는 전체 덮어쓰기 |
| `mkdir /data/output` | `mkdir -p /data/output` |
| POST로 리소스 생성 | idempotency key 사용, 또는 PUT + 고유 식별자 |

## 9. 말없는 fallback 금지

**에러 발생 시 조용히 대체 동작으로 넘어가지 마라.**

- 빈 배열 반환, 기본값 주입, 로그만 찍고 계속 진행 금지.
- 에러는 명시적으로 전파하거나, 호출자가 실패를 인지할 수 있게 하라.
- 빈 catch 블록 금지.
- 정당한 fallback(캐시 미스 → DB 조회 등)에도 반드시 로그나 메트릭을 남겨라.

## 10. 근본 원인 우선
**증상이 아니라 원인을 고쳐라.**

디버깅 시:

- 에러 메시지 키워드만 보고 우회 코드를 추가하지 마라. 메시지가 가리키는 실제 원인을 추적하라.
- 증상이 사라졌다고 작업을 종료하지 마라. "왜 사라졌는가"가 설명되어야 한다.
- 같은 버그가 다른 곳에도 있는지 확인하라. 한 군데서 나타났다면 다른 곳에 잠복할 가능성이 있다.

판단 기준: "이 수정의 메커니즘을 한 문장으로 설명할 수 있는가?" 못 하면 아직 원인을 모른 채 패치한 것이다.

# code-problem-doc

코드베이스에서 버그·설계 결함·구조적 문제를 **조사**하고, 그 결과를
**비개발자(의사결정자·기획자·리뷰어)도 스스로 판단할 수 있는 문제 리포트**로
정리하는 skill이다. 산출물은 Markdown 문서 한 개.

## 언제 쓰나 / 언제 안 쓰나

**쓴다**: "이 부분에서 문제가 난다", "왜 안 되는지 조사해서 정리해줘",
"결함을 문서로 만들어줘" 같은 **결함 조사 + 문서화** 요청. `/code-problem-doc`로
명시 호출됐을 때만.

**안 쓴다**: 단순 코드 수정, 코드 리뷰, 기능 설명 문서, README, 아키텍처 개요.
결함을 실제로 **고치는** 작업(이 skill은 조사·문서화까지만).

## 이 skill이 지키는 7가지 특징

이 순서와 원칙을 문서에 그대로 재현한다.

1. **병렬 서브에이전트 조사** — 문제 영역을 나눠 동시에 조사한다.
2. **검증 수준 구분** — 실증 / 코드 확인 / 부재 확인을 정직하게 나눈다.
3. **원인 분류** — 각 문제를 코드 / 프롬프트 / 설정 / 데이터 층위로 라벨링한다.
4. **easy-answer 서술** — 개발 지식 없어도 읽히게. 용어는 등장 즉시 풀이.
5. **클릭 이동 코드 링크** — 코드 위치를 클릭하면 그 줄로 가게 한다.
6. **수정안 트레이드오프** — 문제마다 본질적으로 다른 2~3개 방법 + 좋은 점/걱정.
7. **정직성** — 실증/추측 구분, 미확인 명시, 버그인지 의도인지 모르면 단정 금지.

## 워크플로우 (5단계)

### 1단계 — 병렬 서브에이전트로 조사

문제 영역을 **2~4개로 나눠** `general-purpose` 서브에이전트를 **한 메시지에서
동시에** dispatch한다(병렬). 각 에이전트에 아래를 반드시 지시한다:

- **코드 수정 절대 금지, 읽기만.**
- 발견을 두 갈래로 나눠 보고:
  - **(A) 코드에 이미 드러난 문제** — 미구현·누락·엣지케이스·비대칭 등
  - **(B) 코드로는 의도를 알 수 없는 것** — "버그인지 설계인지"처럼 원작성자만
    답할 수 있는 지점
- 모든 사실에 **`파일:라인` 인용**.
- **추측은 "추측"이라고 명시.** 확인 안 한 걸 확인한 것처럼 쓰지 말 것.
- 최종 메시지가 그대로 문서 재료가 된다고 알려줄 것.

영역을 나누는 기준 예: 생성 경로 / 수정 경로 / 배포 경로 / 검증(테스트)·관측.
문제가 어디서 나는지 아직 모르면, 먼저 가벼운 정찰(파일 목록·grep) 후 나눈다.

### 2단계 — 검증 수준 구분(가능하면 실증)

각 문제를 어떻게 확인했는지 등급을 매긴다:

- **실증** — 해당 로직(가능하면 외부 의존성 없는 순수 함수)을 **격리 실행**해
  눈으로 확인. 결함 하나라도 이렇게 실증하면 리포트 신뢰도가 크게 오른다.
- **코드 확인** — 코드를 정독해 "이렇게 될 수밖에 없다"를 논리로 확인.
- **부재 확인** — "있어야 할 검사/코드가 없음"을 확인(없음을 증명).

실증하지 않은 것을 실증처럼 쓰지 않는다. 신뢰도 표에 등급을 그대로 적는다.

### 3단계 — 원인 분류

각 문제의 원인을 층위로 라벨링한다: **코드**(실행 로직) / **프롬프트**(AI에게
주는 지시문) / **설정**(config·스키마) / **데이터**(사전·매핑 테이블). 한 문제가
여러 층위에 걸치면 "코드+프롬프트"처럼 병기한다. 이 구분이 중요한 이유:
프롬프트 수정은 효과가 **확률적**(AI가 안 따를 수 있음)이고, 코드 수정은
**확정적**이다 — 독자가 수정 방향의 성격을 알아야 한다.

### 4단계 — 문서화

`references/document-template.md`의 골격을 채운다. 서술 규칙(아래)을 따른다.

### 5단계 — 정직성 점검

문서를 마치기 전 확인: 추측/사실이 섞이지 않았나? 미확인을 미확인이라 했나?
"버그인지 의도인지" 불확정 항목을 단정하지 않고 "먼저 확인할 것"으로 남겼나?

## 서술 규칙 (easy-answer 준수)

- **독자는 비개발자**. 개발 개념 없이 논리로 이해되게 쓴다.
- **용어는 등장 즉시 풀이.** 예: "런타임(실제 실행되는 순간)".
- **모호어 금지.** "설명서" 같은 뭉뚱그린 말 대신 "프롬프트/코드"로 명확히.
- **원인 종류를 문제마다 표시** (코드/프롬프트/설정).
- **비유는 기본 지양.** 사용자가 원할 때만 허용. 대신 실제 동작 메커니즘으로 설명.
- **단계별 인과**로 "왜 안 되는지"를 풀되, 각 단계에 `[코드 문제]`/`[프롬프트
  문제]` 태그를 붙여 층위를 보이게.

## 코드 링크 규칙

코드 위치는 **클릭하면 그 줄로 이동**하게 만든다.

- 형식: `[파일명:라인](산출문서_기준_상대경로#L라인)`
- 중립 예시: `[auth.ts:42](../src/auth.ts#L42)`
- 경로는 **산출 문서 파일 위치 기준 상대경로**로 계산한다(문서가 `docs/`에 있고
  코드가 `src/`에 있으면 `../src/...`).
- fallback: 환경에서 상대경로 링크가 안 열리면 워크스페이스 루트 기준 경로로
  바꾼다. 링크가 여의치 않으면 최소한 `파일:라인`을 텍스트로 정확히 남긴다.

## 문서 구조

`references/document-template.md`를 골격으로 쓴다. 핵심 뼈대:

1. 상단 안내(무엇에 관한 문서 / 확인한 곳 / 링크 클릭 안내)
2. **먼저 알아둘 기본 개념** — 도메인 용어 풀이 + "코드 vs 프롬프트/설정" 구분
3. **이 조사를 얼마나 믿어도 되는지** — 신뢰도 표(문제 | 확인 방법 | 신뢰도)
4. 문제마다: 원인 종류 → 무슨 일이 일어나나 → 왜(단계별, 태그) → 정확히 어느
   코드(표+클릭링크) → 어떻게 고치나(방법 A/B/C, 좋은 점/걱정) → 추천 + 먼저 확인할 것
5. **전체 한눈에 보기** 표
6. **꼭 기억할 것** — SSoT 이중화, 검증 사각지대, 의도 불확정 같은 함정

구조 예시는 `references/example-report.md`를 참고한다(특정 프로젝트 내용이지만
구조·톤의 완성형 예시).

## 정직성 규칙 (Honesty Rules)

- 실증하지 않은 것을 실증한 것처럼 쓰지 마라. 신뢰도 등급을 그대로 적어라.
- 확인 못 한 것은 "미확인"이라고 말해라. 모르면 모른다고 해라.
- 버그인지 의도된 설계인지 코드만으로 모르면 **단정하지 말고** "먼저 확인할
  것"으로 원작성자 질문을 남겨라.
- 수정안은 **위치와 방향 제안**이다. "적용 전 회귀 테스트 필요"를 명시해라.
- 추측은 반드시 "추측"으로 표기해라.

## Why

결함을 "개발자만 아는 것"으로 두지 않고, **원인·수정 방향·리스크를 비개발자
의사결정자가 스스로 판단**할 수 있게 만드는 것이 목적이다. 그래서 쉬운 서술과
정직한 신뢰도 표기, 그리고 "고칠 때 무엇을 먼저 확인해야 하는지"가 함께 간다.

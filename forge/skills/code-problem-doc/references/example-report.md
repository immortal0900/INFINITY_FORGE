# 코파일럿 복잡 워크플로우 문제 4가지 — 쉽게 정리한 원인과 해결법

> 무엇에 관한 문서인가: 코파일럿으로 **loop·router·parallel이 들어간 복잡한 워크플로우를 만들고·고치고·배포할 때** 생기는 문제 4가지를, 개발 지식이 없어도 이해할 수 있게 정리했습니다.
> 확인한 곳: `workflow-engine/apps/backend/lib/copilot/` 코드를 직접 읽고 정리. 코드는 고치지 않았고, "어디가 문제이고 어떻게 고칠 수 있는지"만 담았습니다.
> 표의 코드 위치(파란 링크)를 클릭하면 실제 코드 줄로 이동합니다.

---

## 먼저 알아둘 기본 개념 6가지

이 여섯 개만 알면 아래 내용이 다 읽힙니다.

- **블록(block)**: 워크플로우를 이루는 작업 한 칸. 예: "메일 보내기", "조건 판단하기".
- **연결(edge, 엣지)**: 블록과 블록을 잇는 선. 이 선이 있어야 다음 블록으로 넘어갑니다. 선이 끊긴 블록은 실행되지 않습니다.
- **연결 지점의 이름(handle, 핸들)**: 선이 블록에서 나오는 자리에 붙는 구분용 이름. "이 선은 어느 길로 가는 선"인지 표시합니다.
- **런타임(runtime)**: 워크플로우가 **실제로 실행되는 순간**. 만드는 시점(그림 그리기)과 다릅니다.

이 문서에서 **문제의 원인은 딱 두 종류**로 나뉩니다. 이 둘을 구분하는 게 핵심입니다.

- **프롬프트(prompt)**: 코파일럿(AI)에게 **사람이 글로 써준 지시문**. "이럴 땐 이렇게 만들어라"를 말로 적은 것. → 프롬프트를 고치는 건 "AI에게 다르게 시키는 것"이라 **효과가 확률적**입니다(AI가 안 따를 수도 있음).
- **코드(code)**: 프로그램이 **실제로 실행하는 로직**. 프롬프트와 완전히 다른 층위입니다. → 코드를 고치는 건 "동작을 확정적으로 바꾸는 것"입니다(무조건 그렇게 됨).

**네 문제의 공통점**: 전부 **만들 때는 정상처럼 보이는데, 실제 실행(런타임)에 가서야 터집니다.** 한쪽만 처리되고 그 짝이 안 따라오는 "어긋남"이 원인입니다.

## 이 조사를 얼마나 믿어도 되는지 (정직하게)

| 문제 | 어떻게 확인했나 | 신뢰도 |
|---|---|---|
| #2 router | 실제 함수를 **직접 실행**해서 눈으로 확인 | 확실 (실증) |
| #1 parallel/loop | 코드를 읽어 논리 확인 (항상 발생 확정) | 확실 (코드 확인) |

# 문제 #2 — router(경로 선택 블록)가 두 종류인데 하나만 완성돼 있음

**원인 종류: 프롬프트 + 코드 (둘 다)**

### 무슨 일이 일어나나
router 블록을 만들면 실제 실행할 때 갈림길에서 멈춥니다. 특히 **한국어로 "라우터/경로분기/라우팅"이라고 요청할 때** 잘 생깁니다.

### 왜 이런 일이 생기나 (단계별)
1. router에는 **옛날 종류(`router`)와 새 종류(`router_v2`)** 두 개가 있는데, 선에 이름 붙이는 규칙이 서로 다릅니다.
   - 옛날 종류: 이름 = "도착할 블록의 번호"
   - 새 종류: 이름 = "미리 정해둔 경로의 번호"
2. **[프롬프트 문제]** 코파일럿에게 주는 프롬프트에는 옛날 종류만 적혀 있습니다. 새 종류와 그 "경로 목록"은 한 글자도 안 적혀 있고, 오히려 새 방식 이름은 *쓰지 말라는 금지 예시*로 박혀 있습니다.
3. **[코드 문제]** 새 종류가 필요로 하는 "경로 목록"을 채워주는 코드가 아예 없습니다. (코드 전체를 뒤졌지만 없음 = 확정)
4. **[코드 문제]** 그런데도 한국어 요청은 새 종류로 만들어집니다. 한국어 별칭 사전(코드 안의 데이터)이 "라우터"를 새 종류로 연결해두었기 때문입니다.
5. 결과: 새 종류 블록이 태어나는데, 선 이름도 옛날 방식(혹은 방치)이고 경로 목록도 비어 있어서 → **런타임에 "어디로 보낼지 못 찾고" 갈림길이 죽습니다.**

### 어느 종류로 만들어지는지 (실제로 실행해서 확인)
| 이렇게 요청하면 | 이런 블록이 됨 | 실행되나 |
|---|---|---|
| 영어/규칙 기반 요청 | 옛날 `router` | ✅ 정상 작동 |
| 한국어 "라우터/경로분기/라우팅" | 새 `router_v2` | ❌ 갈림길 죽음 |
| "router_node" 등 변형 표현 | 새 `router_v2` | ❌ 갈림길 죽음 |

> 함정: 품질 검사(eval)가 옛날 종류와 새 종류를 **같은 것으로 취급**해서, 이 문제가 자동 테스트에서도 안 걸립니다. 이유는 채점 코드 안의 별칭 표에 `router → router_v2`가 박혀 있어([block-type.ts:57-61](../workflow-engine/apps/backend/lib/copilot/eval/scorers/block-type.ts#L57)), **채점 전에 두 이름을 하나로 통일**하기 때문입니다.

### 정확히 어느 코드
| 위치 | 종류 | 무슨 문제 |
|---|---|---|
| [edit-workflow/index.ts:519](../workflow-engine/apps/backend/lib/copilot/tools/server/workflow/edit-workflow/index.ts#L519) | 코드 | 선 이름을 "옛날 종류일 때만" 붙임. 새 종류는 방치 |
| [intent-graph/extract.ts:923](../workflow-engine/apps/backend/lib/copilot/intent-graph/extract.ts#L923) | 프롬프트 | 옛날 종류만 가르침, 새 종류는 금지 예시로 명시 |
| (부재) 코드 전체 | 코드 | 새 종류의 "경로 목록"을 채우는 코드가 **없음** |
| [korean-synonyms.ts:41](../workflow-engine/apps/backend/lib/copilot/intent-pipeline/korean-synonyms.ts#L41) | 코드(데이터) | "라우터"를 새 종류로 연결 → 한국어 요청이 새 종류로 감 |
| [eval/scorers/block-type.ts:57](../workflow-engine/apps/backend/lib/copilot/eval/scorers/block-type.ts#L57) | 코드(검사) | 별칭 표가 옛날/새 종류를 통일 → 검사에서 안 걸림 |

### 어떻게 고치나 (세 가지 방법)

**방법 A — 옛날 종류로 통일 (코드 수정, 새 종류가 안 만들어지게 막기)**
- 한국어 별칭 사전에서 "라우터"의 새 종류 연결을 떼고, 변형 표현도 옛날 종류로 가게 함.
- 좋은 점: 옛날 종류는 **이미 모든 게 완성돼 있어서**(프롬프트·선 이름·검사 다 있음) 가장 적은 수정으로 문제가 사라짐. 코드 수정이라 **확정적**.
- 걱정되는 점: 옛날 종류는 "구버전" 딱지가 붙어 있어, 나중에 없어지면 다시 손봐야 함.

**방법 B — 새 종류를 제대로 완성하기 (코드 + 프롬프트 둘 다 수정)**
- 없는 "경로 목록 채우는 코드"를 새로 만들고, 선 이름도 새 방식으로 붙이고, 프롬프트도 새 종류용으로 다시 씀.
- 좋은 점: 최신 버전에 맞춰짐. 미래지향적.
- 걱정되는 점: 손댈 곳이 코드 4곳+프롬프트로 가장 많고, 새 종류는 원래 "AI가 알아서 길을 고르는" 블록이라 경로 설명을 코드로 자동 생성하기가 어렵습니다. 되돌리기 비용이 큼.

**방법 C — 프롬프트만 고치기**
- "라우팅은 옛날 종류를 써라"라고 프롬프트에 명시.
- 좋은 점: 코드를 안 건드려서 즉시 되돌릴 수 있음.
- 걱정되는 점: 프롬프트는 **확률적**이라 100% 못 막습니다. 코드(별칭 사전)가 여전히 새 종류로 끌 수 있어 완전한 해결은 아님.

**추천**: 지금 당장 막으려면 **A**(코드 수정, 옛날 종류가 유일하게 완성된 길). 회사가 새 종류를 정식으로 밀기로 했다면 **B**(별도 작업으로). 어느 쪽이든 **eval의 별칭 표(코드)도 함께 손봐야** 검사가 결함을 잡습니다.
**먼저 확인할 것**: 옛날 종류가 곧 없어질 예정인지. 없어질 거면 A 대신 B로 가야 합니다. (이건 코파일럿 밖 영역이라 확인 못 함)

# 문제 #1 — parallel/loop를 만들 때 설정값이 저장 안 됨 (항상 발생)

**원인 종류: 주로 코드 (프롬프트도 부차적으로 관련)**

### 무슨 일이 일어나나
"A·B·C 세 은행에 동시에 조회" 같은 parallel(병렬, 여러 개를 동시에 처리)을 만들면, **동시에 처리할 대상 목록(A·B·C)이 사라지고** 기본값으로 바뀝니다. loop(순차 반복)도 반복 방식 설정이 누락됩니다. **가끔이 아니라 프롬프트대로 만들면 매번(100%) 발생합니다.**

### 왜 이런 일이 생기나 (단계별)
1. 코파일럿이 parallel 블록을 만들면서 대상 목록(A·B·C)을 설정값으로 채웁니다.
2. **[코드 문제]** 블록을 새로 만드는 코드는, 이 설정값을 저장하는 동작을 `nestedNodes`(자식 블록을 함께 지정하는 항목)가 있을 때만 합니다.
3. **[프롬프트 문제]** 코파일럿 프롬프트는 `nestedNodes`를 안 씁니다. "부모 따로, 자식 따로 만들고 `parentId`로 잇기"라서 `nestedNodes`가 늘 비어 있습니다. 프롬프트·도구 설명 어디에도 `nestedNodes`라는 단어가 없습니다.
4. 결과: 조건이 절대 참이 안 돼 저장 코드가 **한 번도 실행 안 됨** → 설정값(A·B·C)이 매번 사라져 기본값으로 떨어집니다.

**중요 반전 — `nestedNodes`는 설정값과 아무 상관이 없습니다.** 저장 코드는 자식 정보를 한 번도 안 쓰고 부모의 입력값만 봅니다. 즉 **자식과 무관한 코드가 "자식이 있을 때만" 실행되게 잘못 묶여 있는 것**(코드 문제)입니다.
> 결정적 증거: 블록을 **고칠 때(수정)**는 같은 저장 코드가 `nestedNodes` 없이 입력값 기준으로 작동합니다([operations.ts:1610](../workflow-engine/apps/backend/lib/copilot/tools/server/workflow/edit-workflow/operations.ts#L1610)). 그래서 수정은 정상인데 신규 생성만 깨집니다.

### 정확히 어느 코드
| 위치 | 종류 | 무슨 문제 |
|---|---|---|
| [operations.ts:1980](../workflow-engine/apps/backend/lib/copilot/tools/server/workflow/edit-workflow/operations.ts#L1980) | 코드 | 새로 만들 때: 설정 저장이 `nestedNodes` 조건에 잘못 갇힘 |
| [operations.ts:1610](../workflow-engine/apps/backend/lib/copilot/tools/server/workflow/edit-workflow/operations.ts#L1610) | 코드 | 고칠 때: 조건 없이 입력값 기준 저장 (→ 정상, 신규와 비대칭) |
| [scenario-mapping.ts:424](../workflow-engine/apps/backend/lib/copilot/prompts/scenario-mapping.ts#L424) | 프롬프트 | "부모+자식 따로" 방식만 지시, `nestedNodes` 미사용 |
| [copilot-tools.ts:14](../workflow-engine/apps/backend/lib/copilot/copilot-tools.ts#L14) | 코드(스키마) | 도구 스키마에 `nestedNodes` 자체가 정의 안 됨 |

### 어떻게 고치나 (세 가지 방법)

**방법 A — 만들 때와 고칠 때를 똑같이 맞추기 (코드 수정, 권장)**
- 새로 만드는 코드에서 **설정 저장 부분만** `nestedNodes` 조건에서 떼어내, 고칠 때와 똑같이 입력값 기준으로 저장. (자식을 *생성*하는 별도 코드는 `nestedNodes`가 계속 필요하므로 그 부분은 그대로 둠)
- 좋은 점: 가장 적은 수정, 프롬프트를 안 건드림, 부작용 거의 없음(자식 없는 빈 컨테이너도 올바르게 설정됨). 코드 수정이라 확정적.
- 걱정되는 점: 지금처럼 조건을 둔 게 일부러 그런 건지 먼저 확인.

**방법 B — 프롬프트가 `nestedNodes`를 쓰게 하기 (프롬프트 수정, 비권장)**
- "컨테이너는 자식을 `nestedNodes`로 함께 지정해라"라고 프롬프트·스키마를 개편.
- 걱정되는 점: 스키마에 `nestedNodes`가 정의조차 안 돼 있고, 기존 "따로 만들기" 방식과 충돌. 설정값 문제를 고치는 데 자식 정보는 필요 없으므로 **과한 변경**. 게다가 프롬프트라 확률적.

**방법 C — 블록 만드는 코드의 뿌리에서 처리 (코드 수정)**
- 블록 생성 공통 지점에서 컨테이너 설정을 항상 저장.
- 걱정되는 점: 모든 블록 생성에 영향 → 부작용 위험 큼.

**추천**: **A**(코드 수정, 설정 저장만 조건에서 분리, 고칠 때와 대칭). 단 지금 조건을 둔 게 의도인지 먼저 확인.

# 문제 #3 — 복잡한 워크플로우를 고칠 때 현재 상태를 못 봄

**원인 종류: 순수 코드 (프롬프트는 정상)**

### 무슨 일이 일어나나
블록이 많은(대략 8~10개 이상) 복잡한 워크플로우를 고쳐달라고 하면, 코파일럿이 **기존 내용을 못 본 채로 고쳐서** 멀쩡한 블록을 잘못 건드리거나 지웁니다. 단순한 워크플로우는 문제없습니다.

### 왜 이런 일이 생기나 (단계별)
1. 복잡한 워크플로우를 고치려면 코파일럿은 먼저 **지금 상태 전체를 읽어와야** 합니다. 이를 위해 "현재 상태를 줘"라고 도구를 호출하고 서버가 응답을 돌려주는 **왕복(요청→응답 한 바퀴)**을 합니다.
2. **[코드 문제]** 서버 응답을 AI에게 전달할 때, 너무 길면(8천 자 초과) 잘라냅니다. (비용 절약을 위한 정상 동작)
3. **[코드 문제]** 자를 때 "중요한 건 남기자"는 규칙이 있는데, 이 규칙은 **특정 이름(`workflowState` 또는 `data`)으로 된 것만** 남깁니다.
4. **[코드 문제]** 그런데 워크플로우 전체 내용은 **`userWorkflow`라는 다른 이름**으로 오기 때문에, 남길 목록에 안 들어가 **통째로 버려집니다.**
5. 결과: AI는 "블록이 N개 있다"는 요약만 받고 실제 내용을 못 봄 → 못 본 채로 고쳐서 망가뜨립니다.

**핵심**: 자르는 것 자체는 정상입니다. 문제는 **자를 때 `userWorkflow`를 남길 목록에서 빠뜨린 코드**입니다. 프롬프트는 잘못이 없습니다. 그리고 워크플로우가 클수록 확실히 잘리므로 **복잡할수록 더 자주 망가집니다.**

### 정확히 어느 코드
| 위치 | 종류 | 무슨 문제 |
|---|---|---|
| [copilot-tool-loop.ts:167](../workflow-engine/apps/backend/lib/copilot/copilot-tool-loop.ts#L167) | 코드 | 자를 때 `workflowState`/`data`만 남기고 `userWorkflow`는 버림 |
| [get-user-workflow.ts:111](../workflow-engine/apps/backend/lib/copilot/tools/server/workflow/get-user-workflow.ts#L111) | 코드 | 워크플로우를 `userWorkflow` 이름으로 반환 → 위에서 버려짐 |
| [openai-streaming.ts:263](../workflow-engine/apps/backend/lib/copilot/openai-streaming.ts#L263) | 코드 | **똑같은 자르기 코드의 복사본이 하나 더 있음** (아래 주의 참고) |

> ⚠️ 주의(함정): 자르는 코드가 **두 벌** 존재합니다. 한 곳만 고치면 AI 종류(OpenAI vs Claude)에 따라 동작이 달라집니다. **두 곳 다** 고쳐야 합니다.

### 어떻게 고치나 (세 가지 방법)

**방법 A — 남길 목록에 `userWorkflow` 추가 (코드 수정, 임시 완화)**
- 자르는 코드 두 곳에 "`userWorkflow`도 남겨라"를 추가.
- 좋은 점: 변경이 가장 작음.
- 걱정되는 점: 8천 자 한도 안에서만 살아남으므로, **정작 문제인 큰 워크플로우는 여전히 잘립니다.** 근본 해결 아님. 두 곳 동시 수정 필수.

**방법 B — 현재 상태를 처음부터 함께 넘겨주기 (코드 수정, 근본 해결·권장)**
- 위 1번의 "왕복"을 없앱니다. 코파일럿이 도구로 따로 물어보게 하지 말고, 워크플로우 전체 내용을 **처음부터 자르지 않고 AI에게 함께 넘깁니다.** 마침 프롬프트는 이미 "AI에게 주어지는 현재 상태를 읽어라"라고 돼 있어서, **프롬프트를 안 바꿔도 됩니다.** 그리고 이 "자르지 않고 넘기는 방식"은 이미 다른 기능(@현재워크플로우 첨부)에서 **정상 작동이 검증**돼 있습니다.
- 좋은 점: 문제의 뿌리(왕복 + 8천 자 한도)가 사라짐. 큰 워크플로우도 온전.
- 걱정되는 점: 항상 넣으면 비용이 커지므로 **복잡한 수정일 때만** 넣도록 조건을 걸어야 함.

**방법 C — 자르는 한도를 늘리기 / 압축 (코드 수정)**
- 8천 자 한도를 올리거나, 워크플로우를 짧게 압축해서 전달.
- 좋은 점: 간단.
- 걱정되는 점: 한도를 올리면 무관한 것까지 커져 비용 증가. 압축은 또 다른 자르기라 결국 같은 문제 반복.

**추천**: **B**(코드 수정, 뿌리를 없앰, 검증된 방식 재사용). A는 임시방편이고 큰 워크플로우엔 안 통함.

### 고민해볼 사항
- 현재 사용자가 코파일럿에게 수정을 요청할시 도구(get_user_workflow) 호출해서 현재 워크플로우를 파악하는데 이 과정은 항상 필요한 과정이니 불필요한 왕복(요청 응답)을 줄이고 맨처음 프롬프트에 아예 현재 워크플로우를 글자수 제한 없이 던지는 것이 효율적임.

# 문제 #4 — 망가진 워크플로우도 배포됨

**원인 종류: 순수 코드 (프롬프트는 무관)**

### 무슨 일이 일어나나
선이 끊긴 블록이 있어도 **배포가 그냥 성공**합니다. 실제 실행할 때서야 끊긴 데서 멈춥니다. 사용자는 "배포 성공"만 보고 정상이라 믿습니다.

### 왜 이런 일이 생기나 (단계별)
1. 워크플로우가 제대로 돌려면 블록끼리 선이 빠짐없이 이어져 있어야 합니다.
2. **[코드 문제]** 그런데 배포할 때는 "선이 다 이어졌나"를 검사하지 않습니다. 권한과 로그인 여부만 봅니다.
3. **[코드 문제]** 사실 그 검사 기능은 이미 만들어져 있는데, 배포 단계에서 부르지 않습니다.
4. 결과: 끊긴 워크플로우도 배포 성공 → 실행 때 멈춥니다. 문제가 배포가 아니라 한참 뒤 실행 시점에 드러나 원인 찾기도 어렵습니다.

### 정확히 어느 코드
| 위치 | 종류 | 무슨 문제 |
|---|---|---|
| [deploy-workflow.ts:124](../workflow-engine/apps/backend/lib/copilot/tools/server/workflow/deploy-workflow.ts#L124) | 코드 | 배포 시 권한·로그인만 검사, 정합성 검사 없음 |
| [edit-workflow/index.ts:940](../workflow-engine/apps/backend/lib/copilot/tools/server/workflow/edit-workflow/index.ts#L940) | 코드 | 정합성 검사 기능은 **이미 있음** (배포 때 안 씀) |

### 어떻게 고치나 (세 가지 방법)

**방법 A — 배포 전에 검사해서, 문제 있으면 막기 (코드 수정)**
- 이미 있는 검사 기능을 배포 직전에 불러서, 끊긴 블록·미완성 갈림길이 있으면 **배포를 거부**.
- 좋은 점: 기존 기능 재사용. 실행 때 터질 문제를 배포 때 미리 잡음.
- 걱정되는 점: 지금까지 배포되던 게 갑자기 막힐 수 있음 → 정책 결정 필요.

**방법 B — 막지는 않고 경고만 하기 (코드 수정)**
- 배포는 허용하되 "이 상태로는 실행 시 실패할 수 있음"을 강하게 알림.
- 좋은 점: 기존 사용에 지장 없음.
- 걱정되는 점: 사용자가 무시하면 문제 지속.

**방법 C — 화면에서 배포 버튼을 잠그기 (프론트 코드 수정)**
- 문제가 있으면 배포 버튼을 못 누르게 함.
- 좋은 점: 배포 전에 예방.
- 걱정되는 점: 화면(프론트) 영역이라 백엔드 직접 호출은 못 막음.

**추천**: **A와 B는 정책 문제**(안전 우선이냐, 빠른 배포 우선이냐). 코드로는 A가 가능하고 기능도 이미 있습니다. **원작성자에게 정책을 확인한 뒤** 결정.

## 꼭 기억할 것
1. **원인이 프롬프트냐 코드냐를 구분하세요.** 프롬프트 수정은 효과가 확률적(AI가 안 따를 수 있음)이고, 코드 수정은 확정적입니다. #3·#4는 순수 코드 문제라 코드로만 고치면 되고, #2는 프롬프트+코드가 얽혀 있어 한쪽만 고치면 부족합니다.
2. **#3의 자르기 코드는 두 벌**입니다([copilot-tool-loop.ts:167](../workflow-engine/apps/backend/lib/copilot/copilot-tool-loop.ts#L167), [openai-streaming.ts:263](../workflow-engine/apps/backend/lib/copilot/openai-streaming.ts#L263)). 고칠 때 한 곳만 고치면 AI 종류마다 동작이 달라집니다.
3. **품질 검사(eval)가 #2를 못 잡습니다.** 채점 코드의 별칭 표([block-type.ts:57](../workflow-engine/apps/backend/lib/copilot/eval/scorers/block-type.ts#L57))가 옛날/새 router를 같은 것으로 통일하기 때문. #2를 고칠 때 eval도 같이 강화해야 합니다.
4. **#1·#3·#4는 "버그인지 일부러 그런 건지" 코드만으론 확정 못 합니다.** 그래서 각 문제마다 "먼저 확인할 것"을 붙였습니다. 고치기 전에 원작성자 확인이 먼저입니다.
5. 여기 적은 해결법은 **위치와 방향 제안**입니다. 실제로 적용하기 전에 각 방법이 다른 데를 망가뜨리지 않는지 테스트로 확인해야 합니다.
